# Architecture

**Analysis Date:** 2026-02-25

## Pattern Overview

**Overall:** Layered architecture with SDK generation layer, Terraform plugin SDK layer, and Snowflake API integration layer.

**Key Characteristics:**
- Code generator-driven SDK (`pkg/sdk`) that produces generated clients for Snowflake SQL operations
- Terraform Plugin SDK v2 provider (`pkg/provider`) wrapping SDK operations
- Resource and Data Source definitions mapping Terraform concepts to SDK operations
- Schema definitions generated from SDK or manually maintained
- Two-phase entry: TFv5->v6 upgrade server transitioning to v6 plugin framework
- Comprehensive acceptance testing framework with test helpers and fixtures

## Layers

**SDK Layer (Generated):**
- Purpose: Auto-generated Snowflake API client wrapping SQL execution
- Location: `pkg/sdk/`
- Contains: Type definitions, interface implementations, SQL builders, DTO generators
- Depends on: `gosnowflake` driver, internal utilities
- Used by: Resources, datasources, tests
- Pattern: Interface-based design with `<Resource>s` interfaces (e.g., `Accounts`, `Databases`)

**Schema Layer:**
- Purpose: Terraform Plugin SDK schema definitions for resources and data sources
- Location: `pkg/schemas/`
- Contains: Mostly generated schema definitions (marked with `// Code generated by...` comment), some manual schemas
- Depends on: `hashicorp/terraform-plugin-sdk/v2/helper/schema`
- Used by: Resources, datasources
- Pattern: Schema objects defining field types, validation, computed/optional/required properties

**Provider Layer:**
- Purpose: Terraform provider bootstrap and configuration
- Location: `pkg/provider/provider.go`
- Contains: Provider schema definition, resource/datasource mapping, authentication setup
- Depends on: Resources, datasources, SDK
- Used by: Entry point via main.go
- Pattern: Factory functions `getResources()`, `getDataSources()`, `ConfigureProvider()`

**Resources Layer:**
- Purpose: Terraform resource implementations (CRUD operations)
- Location: `pkg/resources/`
- Contains: Individual resource files (`<resource_name>.go`), helpers, state upgraders
- Depends on: SDK, schemas, helpers
- Pattern: CRUD pattern - `Create`, `Read`, `Update`, `Delete` context functions
- Structure: Each resource typically has schema definition and CRUD implementation

**DataSources Layer:**
- Purpose: Terraform data source implementations (read-only queries)
- Location: `pkg/datasources/`
- Contains: Individual datasource files, filtering/aggregation logic
- Depends on: SDK, schemas, helpers
- Pattern: Read-only implementation using SDK Show/List operations

**Helper Layer:**
- Purpose: Shared utilities for resource/datasource implementations
- Location: `pkg/resources/resource_helpers_*.go`, `pkg/resources/helpers.go`
- Contains: Create/Read/Update helpers, identifier encoding, state transformation
- Depends on: SDK
- Used by: Resources, datasources
- Pattern: Builder-pattern helpers for attribute processing

**Acceptance Testing Layer:**
- Purpose: Integration tests against real Snowflake account
- Location: `pkg/testacc/`
- Contains: Setup, fixtures, test helpers, individual test files
- Depends on: SDK, resources, datasources
- Pattern: Acceptance test framework using Terraform plugin testing utilities

## Data Flow

**Provider Configuration Flow:**

1. `main.go` entry point
2. Loads old provider from `pkg/provider/provider.go` via `Provider()`
3. Wraps with v5->v6 upgrade server converter
4. Multiplexes with v6 mux server
5. Configures authentication via `ConfigureProvider()` → creates SDK client
6. Client injected into resource/datasource metadata context

**Resource CRUD Flow:**

1. User specifies resource in Terraform configuration
2. Provider router dispatches to resource implementation file (e.g., `account.go`)
3. Schema retrieved from `pkg/schemas/<resource>_gen.go` (generated)
4. Operation triggered: Create/Read/Update/Delete
5. Resource implementation calls SDK layer methods
6. SDK builds SQL DDL/DML statements
7. Executes via gosnowflake driver
8. Results parsed and mapped back to Terraform state
9. State written via `d.Set()` or returned as diagnostics

**DataSource Read Flow:**

1. User references datasource in Terraform configuration
2. Provider routes to datasource (e.g., `database.go`)
3. Datasource Read function called
4. SDK Show/List operation invoked (e.g., `client.Databases.ShowByID()`)
5. Results wrapped into schema structure
6. State populated via `d.Set()` calls

**Identifier Flow:**

1. Resource identified by FQN (fully qualified name) or composite ID
2. Identifiers parsed using SDK identifier types: `AccountObjectIdentifier`, `DatabaseObjectIdentifier`, `SchemaObjectIdentifier`
3. Encoded/decoded via `helpers.EncodeResourceIdentifier()` for state
4. Parsed back on read via `sdk.Parse<Type>ObjectIdentifier()`

## Key Abstractions

**Resource Identifier Hierarchy:**
- Purpose: Type-safe object reference across different scopes
- Examples: `AccountObjectIdentifier`, `DatabaseObjectIdentifier`, `SchemaObjectIdentifier`
- Pattern: Interfaces with methods like `FullyQualifiedName()`, individual component getters
- Usage: All resource/datasource operations use typed identifiers

**SDK Client Interface Pattern:**
- Purpose: Abstract Snowflake API operations
- Examples: `Accounts`, `Databases`, `Schemas` interfaces in `pkg/sdk/`
- Pattern: Create/Alter/Drop/Show operations with typed options and responses
- Usage: Resources call these methods, never raw SQL

**Schema Builder Pattern:**
- Purpose: Fluent construction of Terraform schema definitions
- Examples: `stringAttributeCreate()`, `intAttributeCreateBuilder()` in `resource_helpers_create.go`
- Pattern: Builder functions accepting schema.ResourceData, returning errors
- Usage: Resources populate SDK request objects from Terraform schema

**State Upgrader Pattern:**
- Purpose: Handle breaking changes in resource schema
- Examples: `*_state_upgraders.go` files for account, database, authentication_policy
- Pattern: Functions implementing `schema.StateUpgrader` with version targeting
- Usage: Automatic state migration when resource ID format changes

**Diff Suppression Pattern:**
- Purpose: Handle non-semantic diffs (case, whitespace)
- Examples: `DiffSuppressStatement()`, `NormalizeAndCompare()` in common.go
- Pattern: Functions matching `schema.DiffSuppressFunc` signature
- Usage: Applied to schema fields via `DiffSuppressFunc` property

**Preview/Experimental Features:**
- Purpose: Flag resources/datasources as preview or experimental
- Location: `pkg/provider/previewfeatures/`, `pkg/provider/experimentalfeatures/`
- Pattern: String constants registered in feature lists, checked at Read time
- Usage: Wraps read operation with feature check, errors if disabled

## Entry Points

**Provider Entry:**
- Location: `main.go`
- Triggers: Terraform provider initialization during `terraform apply/plan`
- Responsibilities: Server setup, version tracking, protocol upgrade/mux

**Provider Configuration:**
- Location: `pkg/provider/provider.go` → `ConfigureProvider()`
- Triggers: After provider schema validation, before resource operations
- Responsibilities: Authentication, SDK client creation, context setup

**Resource Operations:**
- Locations: `pkg/resources/<resource_name>.go` → Create/Read/Update/Delete functions
- Triggers: Terraform state transitions (apply/destroy/refresh)
- Responsibilities: Schema-to-SDK bridging, state management

**DataSource Read:**
- Location: `pkg/datasources/<datasource_name>.go` → Read functions
- Triggers: Terraform reference resolution during plan
- Responsibilities: Query Snowflake, populate computed state

**Acceptance Tests:**
- Location: `pkg/testacc/*_acceptance_test.go`
- Triggers: `make testacc` or `TF_ACC=1 go test ./...`
- Responsibilities: End-to-end testing with real Snowflake account

## Error Handling

**Strategy:** Layered error propagation with context preservation

**Patterns:**

- SDK layer: Returns `error` from SQL execution, validation errors
- Resources/DataSources: Convert SDK errors to Terraform diagnostics via `diag.FromErr()`
- Validation: Pre-execution validation in SDK `validate()` methods or resource code
- Diff suppression: Graceful handling of expected non-semantic differences
- Warnings: Issues flagged via `diag.Diagnostic` with severity indicators
- Safe operations: `ShowByIDSafely()` methods that suppress "not found" errors

## Cross-Cutting Concerns

**Logging:**
- Framework: Go stdlib `log` package
- Usage: Acceptance tests use custom logger (`accTestLog`), resource ops use `log.Printf`
- Pattern: Contextual logging in setup, operation timing, error details

**Validation:**
- Location: `pkg/internal/provider/validators/`, SDK `validate()` methods
- Pattern: Diagnostic validation in schema + SDK pre-execution validation
- Examples: Identifier validation, IP address validation, enum validation

**Authentication:**
- Provider schema fields: `account_name`, `user`, `password`, `private_key`, etc.
- Pattern: gosnowflake config construction from provider schema
- Env vars: Supported via `DefaultFunc: schema.EnvDefaultFunc()`
- Methods: Basic auth, token, private key, SAML, OAuth, Okta

**Tracking:**
- Location: `internal/tracking/`
- Pattern: Context wrapping to track operation type and resource version
- Usage: Wraps CRUD functions for telemetry

**Resource ID Encoding:**
- Pattern: Identifier components joined with `.` separator (not deprecated `|`)
- Location: `pkg/helpers/resource_identifier.go`
- Methods: `EncodeResourceIdentifier()`, `DecodeResourceIdentifier()`
